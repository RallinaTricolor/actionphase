## Design Decisions

### Soft Delete vs Hard Delete

**Decision**: Use **soft delete** (mark as deleted, preserve record)

**Rationale**:
- Consistency with common room messages (already use soft delete)
- Preserves conversation structure and notification history
- Allows for potential audit trails or GM oversight
- Prevents orphaned references in database

**Implementation**:
```sql
-- Soft delete columns
deleted_at TIMESTAMP WITH TIME ZONE
is_deleted BOOLEAN DEFAULT FALSE

-- Deletion query
UPDATE private_messages
SET deleted_at = NOW(), is_deleted = true
WHERE id = $1;
```

### Authorization Model

**Decision**: Only message sender can delete their own messages

**Rules**:
1. ✅ User must be authenticated (JWT token required)
2. ✅ User must be message sender (`sender_user_id` matches token user ID)
3. ✅ Message must exist and not already be deleted
4. ❌ GMs cannot delete player messages (future enhancement if needed)
5. ❌ Conversation participants cannot delete others' messages

**Verification Flow**:
```go
// 1. Get message from database
message, err := qtx.GetPrivateMessage(ctx, messageID)

// 2. Verify ownership
if message.SenderUserID != userID {
    return core.ErrForbidden("You can only delete your own messages")
}

// 3. Check not already deleted (optional)
if message.IsDeleted {
    return core.ErrGone("Message already deleted")
}
```

### Display Behavior

**Decision**: Replace content with "[Message deleted]", preserve metadata

**Visibility**:
- ✅ All conversation participants see "[Message deleted]" text
- ✅ Sender name, character, and timestamp remain visible
- ✅ Message ID and position in conversation preserved
- ❌ Original content is NOT shown to anyone (including sender)

**UI Treatment**:
```tsx
{message.is_deleted ? (
  <span className="italic text-content-tertiary">
    [Message deleted]
  </span>
) : (
  <MarkdownPreview content={message.content} />
)}
```

### Confirmation UX

**Decision**: Require confirmation modal before deletion

**Modal Content**:
- **Title**: "Delete Message?"
- **Body**: "Are you sure you want to delete this message? It will be replaced with '[Message deleted]' for all participants. This action cannot be undone."
- **Actions**: "Cancel" (secondary) | "Delete" (danger)

**Rationale**:
- Prevents accidental deletion
- Sets clear expectations about behavior
- Follows common UX patterns

## Technical Constraints

### Database Constraints

**Schema Limitations**:
- Cannot add NOT NULL constraint to `deleted_at` (must allow NULL for non-deleted messages)
- Must maintain backward compatibility (existing messages have no deletion fields)

**Migration Strategy**:
```sql
-- Up migration
ALTER TABLE private_messages
ADD COLUMN deleted_at TIMESTAMP WITH TIME ZONE,
ADD COLUMN is_deleted BOOLEAN DEFAULT FALSE;

-- Down migration
ALTER TABLE private_messages
DROP COLUMN deleted_at,
DROP COLUMN is_deleted;
```

### API Constraints

**REST Conventions**:
- Use HTTP DELETE verb
- Resource path: `/games/{gameId}/conversations/{conversationId}/messages/{messageId}`
- Return 200 OK on success (not 204, to allow response body with confirmation)
- Return 404 if message not found
- Return 403 if user not authorized
- Return 410 Gone if already deleted (optional, could also return 200 for idempotency)

**Response Format**:
```json
{
  "message": "Message deleted successfully",
  "id": 123
}
```

### Frontend Constraints

**React Query Cache Invalidation**:
- Must invalidate conversation messages query after deletion
- Use query key: `['conversationMessages', conversationId]`
- Consider optimistic update for better UX

**Component Location**:
- Message items likely rendered in `MessageThread` or similar component
- Need to identify exact component during implementation
- Must handle both mobile and desktop layouts

## Open Questions & Future Enhancements

### Current Implementation Scope

**In Scope**:
- ✅ Soft delete by message sender
- ✅ Placeholder text for deleted messages
- ✅ Confirmation modal
- ✅ Real-time update for all participants (via React Query refetch)

**Out of Scope** (Future Enhancements):
- ❌ GM override to delete any message
- ❌ Hard delete (permanent removal from database)
- ❌ Audit log of deleted messages
- ❌ Edit message functionality
- ❌ Undo deletion within time window
- ❌ Bulk delete multiple messages
- ❌ Archive conversation (soft delete all messages)

### Potential Future Decisions

**Question**: Should GMs be able to delete any message in their games?
- **Pro**: Moderation capability, removes inappropriate content
- **Con**: Privacy concerns, trust issues
- **Decision**: Defer to future; implement sender-only first

**Question**: Should we show "edited/deleted by [user]" attribution?
- **Pro**: Transparency, accountability
- **Con**: Clutters UI, may not be needed for private messages
- **Decision**: No attribution for now; just show deleted state

**Question**: Should deleted messages still count toward unread indicators?
- **Pro**: Participant knows something was said even if deleted
- **Con**: Confusing UX, why notify about deleted content?
- **Decision**: Deleted messages do NOT contribute to unread count (handle in read tracking logic)

## Risk Mitigation

### Authorization Bypass Prevention

**Risk**: User could manipulate request to delete others' messages

**Mitigation**:
1. Server-side verification (never trust client)
2. Query message owner before deletion
3. Compare sender_user_id with JWT user ID
4. Comprehensive tests for authorization edge cases
5. Code review with security focus

**Test Coverage**:
```bash
# Attempt to delete another user's message (should fail)
curl -X DELETE \
  -H "Authorization: Bearer $PLAYER1_TOKEN" \
  http://localhost:3000/api/v1/games/2/conversations/1/messages/$PLAYER2_MESSAGE_ID
# Expected: 403 Forbidden
```

### Race Conditions

**Risk**: Message deleted while another user is reading it

**Mitigation**:
1. Use database transactions for consistency
2. Frontend handles 404 gracefully (message deleted before fetch completed)
3. React Query's stale-while-revalidate handles cache updates

### Schema Migration Failures

**Risk**: Migration fails in production, breaks application

**Mitigation**:
1. Test migration in development and staging
2. Create reversible migration (`.down.sql`)
3. Verify backward compatibility (new columns are nullable)
4. Plan rollback procedure

**Rollback Plan**:
```bash
# If migration fails or causes issues
just migrate-down  # Rolls back last migration
just migrate       # Reapply after fixing
```
